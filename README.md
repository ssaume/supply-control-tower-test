# 供需承諾控制塔｜IMPORT9

Build：`2026.07.14-IMPORT9`

這是可直接部署到 GitHub Pages 的純前端互動原型，不需要 Node.js、npm、資料庫或後端服務。

## 本版調整

### 單筆 KEY 單

- 「需求來源」只分為：`客戶`、`內部`。
- 選擇來源後，再顯示對應需求類型：
  - 客戶：客戶預測、客戶 PO、緊急詢單
  - 內部：內部預測、內部需求、計畫庫存
- 業務／提出人、客戶、出貨地皆使用主檔下拉選單。
- 選擇業務或內部提出人後，自動帶入提出部門。
- 需求數量使用數字輸入。
- 需求日期使用日曆選擇。
- 已移除需求說明的預設預測文字。

### 需求狀態

不同需求類型建立後使用不同狀態：

| 需求類型 | 一般狀態 |
|---|---|
| 客戶預測 | 待整合 |
| 客戶 PO | 待確認 |
| 緊急詢單 | 詢單評估 |
| 內部預測 | 待整合 |
| 內部需求 | 待確認 |
| 計畫庫存 | 已確認 |

### 產品接單 LT 與事件審批

每一種產品都有自己的標準接單 LT，例如：

- UPS：45天
- 路由器：28天
- 車載充電器：75天
- 牽引逆變器：90天
- 儲能系統 ESS：120天

當「客戶預測」或「客戶 PO」的需求日期落在產品接單 LT 內：

1. 需求事件狀態自動標記為「待審批」。
2. 系統建立接單 LT 例外審批事件。
3. 審批頁顯示產品 LT、距需求日天數與短少天數。
4. 核准後需求狀態改為「已確認」，進入後續供需計畫。

### 需求來源與關係鏈

- 客戶預測 → 客戶 PO
- 緊急詢單 → 客戶 PO（獨立詢單鏈）
- 內部預測 → 內部需求 → 計畫庫存
- 計畫庫存 → 客戶 PO 消耗

需求總覽的來源篩選只使用「客戶／內部」，關聯圖仍可點擊個別需求類型進行下鑽。

## CSV 欄位

| 欄位 | 必填 | 說明 |
|---|---|---|
| source | 是 | 客戶、內部 |
| demandType | 是 | 與來源相符的需求類型 |
| sku | 是 | 成品料號 |
| factory | 是 | 可生產該成品的工廠代碼 |
| qty | 是 | 需求數量，正整數 |
| demandDate | 是 | YYYY-MM-DD |
| proposer | 建議 | 業務或內部提出人 |
| proposerDept | 建議 | 提出部門 |
| customer | 客戶來源建議 | 客戶名稱 |
| destination | 客戶來源建議 | 國家／城市 |

可使用包內的 `demand-import-template.csv` 測試。

## GitHub Pages 部署

1. 解壓縮部署包。
2. 將包內檔案直接放在 GitHub Pages 發布 branch 的根目錄。
3. 不要只上傳 ZIP，也不要在根目錄外再多包一層資料夾。
4. 到 `Settings → Pages`，選擇 `Deploy from a branch`。
5. 指定發布 branch 與 `/ (root)`。
6. 部署完成後強制重新整理瀏覽器。

成功載入時，左側下方會顯示：
